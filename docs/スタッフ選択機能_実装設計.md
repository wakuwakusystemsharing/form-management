# スタッフ選択機能 実装設計書

作成日: 2026-07-07 / 対象: 予約フォーム（LINE予約・カレンダー表示モード）

---

## 1. この資料について

予約フォームに「スタッフ選択」を追加する機能の設計書です。
**実装前に「6. 決めてほしいこと」への回答が必要です。** 回答をもらってから実装を開始します。

### やりたいこと（要望の整理）

1. フォーム編集の **メニュー構成 → 詳細設定** に「スタッフ選択」を追加。ONにするとフォームにスタッフの単一選択ボタンが出る
2. 各スタッフに、**店舗に連携しているGoogleカレンダーの中から1つ**を紐付けられる
3. フォームの希望日時カレンダーの **〇× が、選択されたスタッフのカレンダーの予定で切り替わる**
4. 予約すると **選択されたスタッフのカレンダーにイベントが作成される**
5. **「指名なし」** を選ぶと、選択した日時に空いているスタッフから**ランダムに1人**選ばれてイベントが作成される
6. LINEで「予約確認」「予約をキャンセル」と送ったときの応答にもスタッフを反映する

## 2. 現状の仕組み（前提知識）

まず「今どう動いているか」を知っておくと、変更点が理解しやすくなります。

| 項目 | 現状 |
|---|---|
| カレンダー | **店舗に1つ**（`stores.google_calendar_id`）。空き状況の読み取りも、予約イベントの書き込みも、この1つのカレンダーに対して行う |
| 空き状況（〇×） | フォームが `GET /api/stores/{storeId}/calendar/availability` を呼ぶ → サーバーが店舗カレンダーの予定一覧を返す → フォームのJSが〇×を描画 |
| 予約作成 | `POST /api/reservations` → DBに予約を保存 → 店舗カレンダーにイベント作成（`createReservationEvent(calendarId, ...)` — **すでにカレンダーIDを引数で受ける作りなので、スタッフのカレンダーIDを渡すだけで書き込み先は変えられる**） |
| カレンダー一覧 | `GET /api/stores/{storeId}/calendar/list` — 店舗が連携しているGoogleアカウントの「書き込み可能なカレンダー一覧」を返すAPIが**すでに存在**（店舗管理画面の「カレンダーを変更」で使用中） |
| LINEの「予約確認」「予約をキャンセル」 | **⚠️ 重要:** Googleカレンダーは読んでいません。**DBの reservations テーブル**から、そのLINEユーザーの予約を取得して表示しています |
| キャンセル | 予約レコードの `google_calendar_event_id` を使って、**店舗カレンダーから**イベントを削除 |

### ⚠️ ここから分かる大事なこと

「予約確認と送られたらスタッフ一覧を出して、選ばれたスタッフのカレンダーを読む」という要望がありましたが、
**予約はぜんぶDBに入っていて、LINEユーザーIDで本人の予約を特定できます**。
予約レコードに「担当スタッフ」を保存しておけば、**お客様に聞き返さなくても**「予約確認」で
担当スタッフ名入りの予約一覧をそのまま返せます（タップの手間が1回減り、実装も安全になります）。
→ 詳細は「5-1」参照。どちらの方式にするかは「6. 決めてほしいこと」で選んでください。

## 3. 全体設計

### 3-1. データ構造

**フォーム設定（`FormConfig` に追加。DBマイグレーション不要 = configはJSONB）**

```typescript
// config.staff_selection
staff_selection?: {
  enabled: boolean;                  // スタッフ選択のON/OFF
  required?: boolean;                // 必須にするか
  allow_no_preference?: boolean;     // 「指名なし」ボタンを出すか
  staff: Array<{
    id: string;                      // ランダム生成ID
    name: string;                    // 表示名（例: 田中）
    calendar_id: string;             // 紐付けるGoogleカレンダーID
    calendar_name?: string;          // 表示用（編集画面で選んだときの名前）
  }>;
};
```

**予約レコード（`reservations` テーブルにカラム追加。マイグレーション必要）**

```sql
ALTER TABLE reservations ADD COLUMN IF NOT EXISTS staff_id TEXT;            -- スタッフID（指名なし時は自動割当されたスタッフ）
ALTER TABLE reservations ADD COLUMN IF NOT EXISTS staff_name TEXT;          -- スタッフ表示名
ALTER TABLE reservations ADD COLUMN IF NOT EXISTS staff_calendar_id TEXT;   -- イベントを作成したカレンダーID
ALTER TABLE reservations ADD COLUMN IF NOT EXISTS staff_no_preference BOOLEAN DEFAULT FALSE; -- 指名なしだったか
```

`staff_calendar_id` を保存するのが**最重要ポイント**です。キャンセル時に
「どのカレンダーからイベントを消すか」が分からなくなるためです（5-2参照）。

### 3-2. 処理の流れ（図）

```
【フォーム表示】
  メニュー選択 → スタッフ選択（田中/佐藤/指名なし）→ 希望日時カレンダー表示
                     │
                     ├ 田中選択 → availability API を「田中のカレンダーID」で呼ぶ → 田中の予定で〇×
                     └ 指名なし → availability API を「全スタッフ」で呼ぶ → 誰か1人でも空いていれば〇

【予約送信】POST /api/reservations
  1. DBに予約保存（staff_id / staff_name / staff_calendar_id も保存）
  2. 指名なしの場合: サーバーで全スタッフの空きを再確認 → 空いている人からランダムに1人決定
  3. 決まったスタッフのカレンダーにイベント作成（既存の createReservationEvent を流用）

【キャンセル】（管理画面 / LINE どちらからでも）
  予約レコードの staff_calendar_id + google_calendar_event_id でイベント削除
```

## 4. 実装フェーズ（この順番で作る）

### Phase 1: 設定まわり（フォーム編集画面）

**やること**
- `src/types/form.ts` に `staff_selection` 型を追加
- `src/lib/form-normalizer.ts` にデフォルト値（enabled: false）を追加
- `MenuStructureEditor.tsx` の詳細設定に「スタッフ選択」セクションを追加
  - ON/OFFトグル、「必須にする」「指名なしを表示」チェックボックス
  - スタッフの追加/削除/並び替え（名前入力 + カレンダー選択ドロップダウン）
  - カレンダー選択肢は**既存の** `GET /api/stores/{storeId}/calendar/list` から取得
  - ⚠️ フォーム編集画面は storeId を知っている必要あり（既存のフォーム編集ページは store 配下なのでOK）

**変更ファイル**: `types/form.ts` / `form-normalizer.ts` / `MenuStructureEditor.tsx`

### Phase 2: フォーム上のスタッフ選択UI + 空き状況の連動

**やること**
- `static-generator-reservation.ts`:
  - スタッフ選択ボタンをレンダリング（ご来店回数と同じ単一選択ボタンデザイン + 指名なしボタン）
  - スタッフを切り替えたら空き状況を再取得してカレンダー再描画（メニュー変更時と同じ流れ）
  - スタッフ選択が必須ONで未選択なら、カレンダーを出さない or 送信時にエラー
- `availability API` の拡張:
  - `?staff_id=xxx` パラメータを追加 → フォーム設定からそのスタッフの calendar_id を引いて予定を取得
  - `?staff_id=all`（指名なし用）→ 全スタッフの予定をまとめて返す（後述の freeBusy 推奨）
  - ⚠️ **セキュリティ**: フォームから任意の calendar_id を直接受け取ってはいけません
    （他人のカレンダーIDを渡して中身を覗けてしまうため）。必ず「フォーム設定に登録済みの
    スタッフID」を受け取り、サーバー側で calendar_id に変換すること

**変更ファイル**: `static-generator-reservation.ts` / `api/stores/[storeId]/calendar/availability/route.ts`

### Phase 3: 予約送信とイベント作成（＋指名なしランダム割当）

**やること**
- `POST /api/reservations` の拡張:
  - リクエストに `staff_id`（または `no_preference: true`）を追加
  - スタッフ指名あり: フォーム設定から calendar_id を引いて `createReservationEvent` に渡す
  - **指名なし**: サーバー側で全スタッフの該当時間帯の空きを確認（Google の freeBusy API を推奨。
    複数カレンダーの空き/埋まりを1回のリクエストで取れる）→ 空いているスタッフから
    `Math.random()` で1人選ぶ → その人のカレンダーへ
  - 予約レコードに staff_id / staff_name / staff_calendar_id / staff_no_preference を保存
  - Googleカレンダーのイベントタイトルに担当を入れる（例: `予約: 山田様（担当: 田中)`）
- DBマイグレーション（3-1のSQL）を staging → 本番の順で実行

**変更ファイル**: `api/reservations/route.ts` / `google-calendar.ts`（freeBusyヘルパー追加）/ マイグレーションSQL

### Phase 4: キャンセルとの連動

**やること**
- キャンセル処理（`PATCH /api/reservations/{id}` と LINE Webhook のキャンセル）で、
  イベント削除先を `store.google_calendar_id` 固定 → **`reservation.staff_calendar_id`（なければ従来どおり店舗カレンダー）** に変更

**変更ファイル**: `api/reservations/[reservationId]/route.ts` / `api/webhooks/line/route.ts`

### Phase 5: LINEの「予約確認」「予約をキャンセル」への反映

**推奨案（シンプル）**: 予約レコードに staff_name が入っているので、既存の予約一覧バブルに
「👤 担当: 田中」の行を1行追加するだけ。聞き返し不要で、タップ回数も増えない。

**要望どおりの案（スタッフに聞き返す）**: 「予約確認」→ スタッフ一覧のボタンを返す →
タップ（postback）→ そのスタッフの予約だけ表示。
⚠️ この案の注意点: LINEの応答は「1度きりのreplyToken」で動いているため、聞き返しには
postback イベントの受信処理を新設する必要があり、工数が増えます。また「自分がどのスタッフで
予約したか覚えていないお客様」は結局全スタッフをタップして探すことになります。

→ **推奨案を先に入れて、聞き返し方式が本当に必要か運用を見て判断**するのがおすすめです。

**変更ファイル**: `api/webhooks/line/route.ts`

### Phase 6: 表示系の仕上げ

- ご予約内容（サマリー）に「担当スタッフ」行を表示
- LINE送信メッセージ（予約完了時）に《担当スタッフ》を追加。指名なしの場合は
  「指名なし（担当: 田中）」のように**割当結果を伝える**（サーバー割当なので、送信後の
  API レスポンスから受け取ってメッセージに入れる）
- 管理画面の予約一覧・予約詳細に担当スタッフ列を表示
- Web予約のメールテンプレートに担当スタッフを追加
- リマインダー（send-reminders）のメッセージに担当スタッフを追加（任意）

## 5. 見落としがちなポイント（重要）

### 5-1. 「予約確認」でスタッフのカレンダーを読む必要はない
2章のとおり、予約確認/キャンセルは**DBから**取得しています。予約レコードにスタッフを
保存すれば聞き返しは不要です（要望どおりの聞き返し方式も作れますが、postback処理の
新設が必要で工数増）。

### 5-2. キャンセル時に「どのカレンダーから消すか」問題
イベントIDだけ保存していても、**どのスタッフのカレンダーのイベントか**が分からないと
消せません。だから予約レコードに `staff_calendar_id` を必ず保存します。
（指名なしの場合も「実際に割り当てられたスタッフのカレンダーID」を保存）

### 5-3. ダブルブッキング（同時予約）対策
2人のお客様が同じ枠を同時に予約すると、〇×表示は間に合いません。
**予約送信時にサーバー側でそのスタッフの該当時間帯を再チェック**し、埋まっていたら
「申し訳ありません、その時間は埋まりました」を返す処理が必須です（指名なしなら他の
空いているスタッフに回せるので、指名なしのほうが実は埋まりにくい）。

### 5-4. 指名なしで全員×だった場合
選択時点では誰か空いていても、送信時に全員埋まっていることがあり得ます。
→ エラーメッセージを返して日時を選び直してもらう仕様にします。

### 5-5. カレンダーの権限
スタッフのカレンダーに書き込むには、**店舗が連携しているGoogleアカウントが、その
カレンダーの「予定の変更権限」を持っている**必要があります。
- 店舗OAuth連携（store_oauth）の店舗: `/calendar/list` が「書き込み可能なカレンダー」だけを
  返すので、その中から選ばせれば安全 ✅
- サービスアカウント方式（system）の店舗: カレンダーが1つしかないため、**スタッフ選択を
  使うには店舗OAuth連携が前提**になります。編集画面では store_oauth 未連携の店舗に
  「Googleカレンダー連携が必要です」と案内を出します
- 運用: スタッフ個人のGoogleカレンダーを使う場合は、そのスタッフが店舗アカウントに
  カレンダーを「変更権限付きで共有」する設定が必要（店舗向け手順書を用意すると親切）

### 5-6. スタッフの休みの表現
スタッフのカレンダーに「終日予定」や「休み」の予定を入れれば、その日は×になります。
＝ シフト管理機能を作らなくても、**カレンダー運用だけでスタッフの休みを表現できます**（周知事項）。

### 5-7. 「同時刻に埋まるイベント数（max_concurrent_events）」の解釈
現在は店舗カレンダーに対する閾値ですが、スタッフ選択ONの場合は**スタッフ1人につき
1件で埋まる**のが自然です（1人が同時に2人施術できないため）。スタッフ選択ON時は
この設定を無視して常に1件=×とする方針を推奨（要確認 → 6章）。

### 5-8. 希望日時モード（第三希望日時選択）との関係
希望日時モードはカレンダー連携なし（イベント作成もスキップ）なので、〇×連動や
自動割当は対象外。スタッフ選択ボタンだけ表示して「希望スタッフ」として記録する形に
なります（対象にするかは 6章で確認）。

### 5-9. パフォーマンス
指名なしの〇×表示は「全スタッフ分の予定」が必要です。スタッフが5人いれば5カレンダー分。
1件ずつ取得すると遅いので、Google Calendar の **freeBusy API（複数カレンダーを1リクエストで
空き照会できる）** を使います。既存の空きキャッシュ（availabilityCache）もスタッフ別に
キーを分ける必要があります。

### 5-10. その他の細かい連動
- 「前回と同じメニューで予約する」: スタッフ選択も復元するか（推奨: する）
- 予約ステータスを cancelled → confirmed に戻す運用がある場合、イベントの再作成は
  現状もしていないので同様（スコープ外として明記）
- プレビュー / local 環境: カレンダー一覧が取れないため、編集画面ではダミー表示、
  フォームは従来どおり営業時間のみで〇×表示
- スタッフ人数の上限: UI的に10人程度までを推奨（ボタンが縦に並びすぎるため）
