## 1. この機能はなにか（概要）

店舗がホットペッパービューティーやエキテンなど**複数の予約媒体**を使っている場合、それぞれの媒体から「予約が入りました」という**通知メール**が店舗宛てに届きます。

この機能は、その通知メールを**弊社が管理する受信用メールアドレスに転送してもらう**ことで、メール内容を自動で読み取り、**店舗のGoogleカレンダーに予約イベントを自動作成**するものです。

```
お客様が媒体で予約
      ↓
媒体 → 店舗のメールに予約通知メールが届く
      ↓（店舗側で自動転送を設定 / 媒体の通知先に直接指定）
弊社の受信用アドレス（店舗ごとに専用アドレスを発行）
      ↓（Resend Inbound が受信 → Webhook で本システムに通知）
本システムがメール本文を解析（媒体別パーサー）
      ↓
店舗のGoogleカレンダーに予約イベントを自動作成
```

**店舗のメリット**: 複数の予約媒体の管理画面を行き来しなくても、**Googleカレンダーを見るだけで全媒体の予約が確認できる**。さらに本システムの予約フォームは店舗カレンダーの予定を見て〇×を出しているため、**媒体経由の予約が入った時間帯は自動的にフォームでも✕になる**（ダブルブッキング防止の副次効果）。

### 参考実装（このフォルダー内の Apps Script）

| ファイル | 内容 | 設計で引き継ぐポイント |
|---|---|---|
| ホットペッパーバージョン | Gmail検索 → 本文の■ラベルを正規表現で抽出 → イベント作成 | 「所要時間目安：◯時間◯分」が本文にあれば所要時間に使える |
| エキテンバージョン | 件名キーワードで絞り込み → 【ラベル】抽出 → 予約番号で重複チェック | 所要時間はメールに無い（固定20分にしていた）→ **所要時間設定が必要**。予約番号による重複防止は必須 |

## 2. 全体アーキテクチャ

### 2-1. メールの受信方法（Resend Inbound）

本システムはすでに送信で Resend を使用している。受信は **Resend の Inbound（受信）機能**を使う。

- 受信専用サブドメイン（例: `inbound.send.wakuwaku-系ドメイン`）に MX レコードを設定
- 店舗ごとに**専用の受信アドレス**を発行: `rsv-{ランダム12文字}@inbound.…`
  - **店舗IDをそのままアドレスにしない**（推測されてスパムを流し込まれるのを防ぐため、ランダムなトークンを使い、DB で店舗と紐付ける）
- メールを受信すると Resend が **Webhook（`email.received`）** で本システムの API を呼び出す
- Webhook には**署名検証**（svix 署名）を必ず入れる（なりすまし対策）

> ⚠️ **Phase 0 で要確認**: 現在の Resend プランで Inbound が使えるか、受信サイズ・通数制限、日本語メール（ISO-2022-JP 等）の扱い。万一使えない場合の代替案は「10. 代替案」を参照。

### 2-2. 処理フロー

```
POST /api/webhooks/inbound-email （Resend Webhook）
  1. 署名検証（不正リクエストは 401）
  2. 宛先アドレス（rsv-xxxx@…）から店舗を特定 → 見つからなければ破棄
  3. 差出人・件名から媒体を判定（ホットペッパー / エキテン / 不明）
  4. 媒体別パーサーで本文から予約情報を抽出
     （予約番号・来店日時・お名前・メニュー・スタッフ・クーポン・電話番号）
  5. 所要時間を決定（優先順位は「3. 所要時間の決め方」）
  6. 重複チェック（store_id + 媒体 + 予約番号 が登録済みならスキップ）
  7. external_reservations テーブルに保存
  8. 店舗のGoogleカレンダーにイベント作成（既存の createReservationEvent を流用）
  9. 解析失敗時も受信記録は必ず残す（管理画面で確認・手動対応できるように）
```

### 2-3. なぜ Apps Script のままではダメか

参考実装（GAS）は動くが、店舗ごとに Gmail・スクリプト・カレンダーIDを手作業で設定する必要があり、**店舗数が増えると運用が破綻**する。本システムに取り込むことで、店舗管理画面から ON/OFF・所要時間設定ができ、受信ログも一元管理できる。

## 3. 所要時間の決め方（この機能の最大の課題）

媒体の通知メールには所要時間が**書いてないことが多い**（エキテンは無い。ホットペッパーは「所要時間目安」がある場合のみ）。カレンダーイベントの長さを決めるため、以下の優先順位で決定する:

```
① メール本文に所要時間の記載があればそれを使う
   （例: ホットペッパーの「所要時間目安：1時間30分」）
② 店舗が設定した「メニュー名キーワード → 所要時間」ルールに一致すればそれを使う
   （例: 「カット」を含む → 60分、「カラー」を含む → 120分。上から順に最初に一致したもの）
③ どれにも当てはまらなければ店舗の「デフォルト所要時間」（初期値 60分）
```

②のルールは**管理画面で店舗ごとに自由に追加・並び替え・削除**できるようにする（後述の管理画面）。

## 4. データベース設計（新規3テーブル）

```sql
-- 店舗ごとの連携設定
CREATE TABLE external_mail_integrations (
  store_id TEXT PRIMARY KEY REFERENCES stores(id) ON DELETE CASCADE,
  enabled BOOLEAN NOT NULL DEFAULT false,
  inbound_token TEXT NOT NULL UNIQUE,        -- 受信アドレスの rsv-{token} 部分
  default_duration_minutes INTEGER NOT NULL DEFAULT 60,
  target_calendar_id TEXT,                    -- 空なら stores.google_calendar_id を使用
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- メニュー名キーワード → 所要時間のルール（上から順に評価）
CREATE TABLE menu_duration_rules (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  store_id TEXT NOT NULL REFERENCES stores(id) ON DELETE CASCADE,
  keyword TEXT NOT NULL,                      -- メニュー名に含まれる文字列
  duration_minutes INTEGER NOT NULL,
  sort_order INTEGER NOT NULL DEFAULT 0,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 受信・解析した外部予約（重複防止と受信ログを兼ねる）
CREATE TABLE external_reservations (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  store_id TEXT NOT NULL REFERENCES stores(id) ON DELETE CASCADE,
  source TEXT NOT NULL,                       -- 'hotpepper' | 'ekiten' | 'unknown'
  reservation_number TEXT,                    -- 媒体の予約番号（重複防止キー）
  reservation_date DATE,
  reservation_time TEXT,
  duration_minutes INTEGER,
  customer_name TEXT,
  customer_phone TEXT,
  menu_text TEXT,
  staff_text TEXT,
  google_calendar_event_id TEXT,              -- 作成したイベントID（キャンセル対応用）
  status TEXT NOT NULL DEFAULT 'created',     -- 'created' | 'parse_failed' | 'skipped_duplicate' | 'cancelled'
  mail_subject TEXT,
  mail_from TEXT,
  raw_body TEXT,                              -- 解析失敗時の調査用に原文保持（一定期間で削除検討）
  error_message TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE UNIQUE INDEX uniq_external_reservation
  ON external_reservations(store_id, source, reservation_number)
  WHERE reservation_number IS NOT NULL AND status IN ('created', 'cancelled');
```

RLS: 店舗管理者は自店舗の行のみ参照可（既存の `store_admins` パターンを踏襲）。書き込みは service role（Webhook）のみ。

## 5. 媒体別パーサー

`src/lib/external-mail/` に媒体ごとのパーサーを置く（追加しやすい構造にする）:

```
src/lib/external-mail/
  ├── index.ts          … 媒体判定（差出人・件名）と振り分け
  ├── parse-hotpepper.ts … ■ラベル形式（参考GASの正規表現を移植）
  ├── parse-ekiten.ts    … 【ラベル】形式（同上）
  └── types.ts           … ParsedReservation 型
```

**媒体判定の例**:

| 媒体 | 差出人 | 件名キーワード |
|---|---|---|
| ホットペッパー | `@salonboard.com` / `@beauty.hotpepper.jp` 等 | 「予約」を含む |
| エキテン | `reserve@ekiten.jp` | 「即予約を受け付けました」等 |

- **転送メール対策**: Gmail 自動転送では差出人が保たれるが、手動転送だと差出人が店舗になる。本文先頭の転送ヘッダ（`---------- Forwarded message ---------` 等）を剥がしてから解析する
- HTMLメールのみの場合はテキスト変換してから解析
- 全角スペース→半角の正規化（参考GASと同じ）
- 解析に失敗しても **`parse_failed` として必ず記録**し、管理画面で本文を確認できるようにする

## 7. 課題と見落としがちなポイント

### 7-1. 所要時間が不明（→ 3章で対応）

### 7-2. ⭐ Gmail自動転送には「確認コード」の壁がある

店舗が Gmail の自動転送を設定すると、Google は**転送先アドレス宛てに確認メール**（確認コード/リンク）を送る。転送先は弊社の受信用アドレスなので、**店舗はそのメールを見られず転送設定を完了できない**。

**対策**: 受信 Webhook で `forwarding-noreply@google.com` からのメールを検知したら、受信ログに目立つ形で表示（確認リンク/コードをそのまま見せる）。店舗はそれを見て確認を完了する。この動線を作らないと**導入がほぼ確実に詰まる**。

### 7-3. 媒体側のメールフォーマット変更リスク

媒体がメール文面を変えるとパーサーが壊れる。対策:
- 解析失敗を `parse_failed` で記録し、原文を保持（調査→パーサー修正が即できる）
- 失敗が続いたら管理者（弊社）に通知する仕組み（Phase 3）

### 7-4. キャンセル・変更の通知

媒体はキャンセル/変更の通知メールも送る。Phase 1 では**新規予約のみ**対応し、キャンセルメールは受信ログに `unknown` として残るだけ（カレンダーは店舗が手動修正）。Phase 3 でキャンセル通知の解析 → `google_calendar_event_id` のイベント自動削除に対応する。

### 7-5. 重複登録の防止

- 媒体の**予約番号**でユニーク制約（同じ予約番号は2回登録しない）
- メールの再送・二重転送・Webhookのリトライにこれで耐える
- 予約番号が取れないメールは日時+名前で近似チェック（Phase 3）

### 7-6. スパム・なりすまし

- 受信アドレスはランダムトークン入りで推測不可
- Webhook は署名検証必須
- 媒体判定に一致しないメールはイベントを作らずログのみ（`unknown`）

### 7-7. 文字コード・HTMLメール

日本の予約媒体は ISO-2022-JP 等のメールもある。Resend 側でデコードされた本文が取れるかを Phase 0 で確認。HTMLのみのメールはタグ除去してから解析。

### 7-8. 既存機能との関係

- **空き状況**: 店舗カレンダーにイベントが入るため、フォームの〇×に自動反映される（メリット）
- **スタッフ選択**: メールの「スタッフ」欄とスタッフ別カレンダーの紐付けは Phase 3（まずは店舗カレンダーに集約）
- **CRM**: 外部予約のお客様を customers に自動登録するのは Phase 3（電話番号で照合）。誤マージのリスクがあるため最初はやらない
- **リマインダー**: 外部予約は `reservations` テーブルに入れないため、LINEリマインダーの対象外（媒体側がリマインドを送るのが普通なので二重送信を避ける意味でも妥当）

### 7-9. 個人情報の取り扱い

メール原文（氏名・電話番号を含む）を DB に保持するため、保持期間を決める（例: 90日で raw_body を削除するジョブ）。プライバシーポリシーへの追記も検討。

## 10. 代替案（Resend Inbound が使えなかった場合）

1. **Cloudflare Email Routing + Workers**: 無料でメール受信 → Worker から本システムの API を呼ぶ。ドメインが Cloudflare 管理なら有力
2. **Gmail + Apps Script ブリッジ**: 弊社管理の Gmail で受信し、GAS が定期的に本システムの API へ POST（参考GASの発展形。運用は増えるが確実に動く）
3. **SendGrid Inbound Parse / Mailgun Routes**: 受信 Webhook の実績が長いサービスを受信専用に併用

いずれの場合も「2-2. 処理フロー」の 2 以降（店舗特定〜イベント作成）は共通で、**受信の入口だけ差し替え**られる設計にしておく。

| 11-2 | 受信サブドメインの正式名称（例: `inbound.send.〜` の具体名） | DNS（MXレコード）を追加できる担当者・権限の確認も含む |
| 11-3 | 実メールサンプルの入手方法 | ホットペッパー / エキテンの**本物の通知メール**が最低数通ずつ必要（パーサーの精度はサンプルの質で決まる）。既存店舗に協力依頼するか、テスト予約で発生させるか |

### Phase 2（パイロット導入前後）で決めるもの

| # | 決めること | 補足 |
|---|---|---|
| 11-4 | パイロット店舗の選定（1店舗） | ホットペッパーまたはエキテンを実際に使っていて、協力的な店舗 |
| 11-5 | 店舗向けセットアップガイドの内容 | 「媒体の通知先を受信アドレスに変更する」or「Gmailの自動転送を設定する」のどちらを推奨手順にするか（7-2 の確認コード動線の説明を含む） |
| 11-6 | 解析失敗の一次対応フロー | 失敗ログを誰が定期確認し、パーサー修正を誰に依頼するか（当面: 店舗→弊社に連絡、の運用でよいか） |

## 12. 機能追加ロードマップ（フェーズ計画）

各フェーズの**やること・完了条件・リリース単位**。1フェーズずつ確実に進める（スタッフ選択機能と同じ進め方）。

### Phase 0: 受信基盤の検証（実装前の確認）— 目安 0.5〜1日

- やること:
  - Resend Inbound の利用可否・料金・制限を確認
  - 受信サブドメインに MX を設定し、テストメールの受信 → Webhook 到達を確認
  - ホットペッパー / エキテンの実メールを転送して、本文（日本語・文字コード・HTML）が正しく取れるか確認
- 完了条件: 実メールの本文が Webhook で正しく受け取れること
- リリース: なし（検証のみ）。**ここで NG なら10章の代替案に切り替えて再検証**

### Phase 1: 受信 → カレンダー登録の基盤 — 目安 2〜3日

- やること:
  - DBマイグレーション（4章の3テーブル + RLS）
  - `POST /api/webhooks/inbound-email`（署名検証・店舗特定・重複防止・受信ログ）
  - ホットペッパー / エキテンのパーサー（参考GASの移植 + 実メールで調整）
  - 所要時間決定ロジック（3章の優先順位）
  - Googleカレンダーへのイベント作成（タイトルは `HPB予約, 山田太郎` 形式 = Q4-A）
- 完了条件: テスト店舗に実メールを転送するとカレンダーにイベントが自動作成される
- リリース: なし（管理画面がまだないため内部確認まで）

### Phase 2: 店舗管理画面 — 目安 1.5〜2日

- やること: 6章の設定画面一式（ON/OFF・受信アドレス表示・デフォルト所要時間・メニュー別所要時間ルール・登録先カレンダー・受信ログ + Gmail転送確認メールの表示）
- 完了条件: 店舗管理者が自力で連携を開始でき、受信状況を確認できる
- リリース: **staging → 本番 → パイロット店舗1店舗に導入**

### Phase 2.5: パイロット運用 — 目安 2週間程度（開発ではなく運用期間）

- やること: 実運用でパーサーの取りこぼし・所要時間ルールの使い勝手・転送設定のつまずきを収集し、微修正
- 完了条件: パイロット店舗で2週間、解析失敗や重複がない（または許容範囲）
- リリース: 問題なければ**一般公開**（全店舗が設定画面から利用可能に）

### Phase 3: 運用改善 — 目安 1〜2日

- やること:
  - キャンセル通知メールの解析 → 作成済みイベントの自動削除（11-9 の判断後）
  - 解析失敗が続いた場合の弊社への通知
  - raw_body の90日自動削除ジョブ（Q6-A。pg_cron で実装、リマインダーと同じ方式）
- リリース: staging → 本番（機能ごとに分割リリース可）

### Phase 4: 拡張 — 必要に応じて

- 対応媒体の追加（楽天ビューティー / minimo 等。11-12 の要望順）
- 汎用パーサー（未知の媒体でもラベルをある程度自動推測して下書き扱いにする）
- 日時変更通知への対応（11-10）
- スタッフ振り分け（メールのスタッフ名 → スタッフ別カレンダー。11-11 の判断後）

### Phase 5: 発展（アイデア段階）

- 外部予約を管理画面の予約一覧・予約分析に統合表示（本システムの予約と横断で見られるように）
- CRM連携: 外部予約のお客様を電話番号で customers に紐付け、来店履歴に統合（誤マージ対策を設計してから）
- 媒体別の予約件数レポート（どの媒体からの予約が多いかの可視化）

> **進め方**: 各フェーズの着手前に「このフェーズを開始してよいか」を確認し、フェーズ完了ごとに動作確認 → 次へ進む。Phase 0 の検証結果によっては 2章のアーキテクチャ（受信の入口）を差し替える。

## 13. 作業分担（どこまで AI に任せられるか）

**凡例**: 🤖 = AI（Claude）に任せられる ／ 👤 = あなた（人間）の操作が必要 ／ 🤝 = AI が手順を用意し、あなたが実行（いつものダッシュボード作業パターン）

### ざっくり全体像

| 作業の種類 | 分担 | 補足 |
|---|---|---|
| コードの実装・テスト・staging プッシュ | 🤖 | 型チェック / ESLint / シミュレーションテストまで込み |
| DBマイグレーション（SQL）の実行 | 🤝 | SQL は AI が用意、実行はあなたが Supabase ダッシュボードで（いつもの手順） |
| Resend ダッシュボードの操作 | 👤 | Inbound の有効化・受信ドメイン追加・Webhook 登録・APIキーの確認 |
| DNS（MXレコード）の追加 | 👤 | ドメイン管理画面での作業。AI が設定値（ホスト名・値・優先度）を具体的に提示 |
| 環境変数の追加（Vercel） | 🤝 | 変数名と値の場所は AI が指示、貼り付けはあなた |
| 実メールサンプルの入手・転送テスト | 👤 | 本物のホットペッパー / エキテンのメールは店舗・実アカウントからしか出せない |
| パーサーの実装・実メールでの調整 | 🤖 | ただし実メールの本文をチャットに貼ってもらう必要あり（個人情報はマスクしてOK） |
| 管理画面 UI の実装 | 🤖 | |
| パイロット店舗との調整・案内 | 👤 | AI は店舗向けセットアップガイドの文面を作成 |
| main への PR 作成・マージ | 👤 | 説明文は AI が毎回用意（いつもの手順） |

### フェーズ別の分担イメージ

| フェーズ | 🤖 AI がやること | 👤 あなたがやること |
|---|---|---|
| Phase 0 | 検証手順書の作成、Webhook 受け口のテスト実装、受信テストの結果判定 | Resend で Inbound 設定、DNS に MX 追加、テストメール送信・実メール転送 |
| Phase 1 | マイグレーションSQL作成、Webhook・パーサー・所要時間ロジックの実装とテスト | SQL 実行（staging → 本番）、Webhook シークレット等の環境変数設定 |
| Phase 2 | 管理画面の実装、店舗向けセットアップガイド文面の作成 | staging での動作確認、PR マージ、パイロット店舗への案内 |
| Phase 2.5 | 失敗ログの原因分析・パーサー修正 | パイロット店舗のフォロー、失敗ログ・実メールの共有 |
| Phase 3以降 | 実装全般（キャンセル対応・削除ジョブ等） | SQL / Edge Function デプロイの実行、動作確認 |

**目安**: 全体の作業量のうち、コードに関わる部分（実装・テスト・修正）はほぼすべて AI に任せられる。あなたの作業は「**外部サービスのダッシュボード操作**（Resend・DNS・Supabase・Vercel）」「**実メールの用意**」「**確認とマージ**」の3種類に集約され、いずれも AI が1ステップずつ手順を案内する。
